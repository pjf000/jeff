项目背景：

张三 / 李四 / 王五 打算共同协作，开发一套博客系统

项目分工：

张三 / 李四 负责文章系统 王五负责评论系统

建立服务端仓库

git虽然是分布式版本管理工具，但为了方便管理，我们需要建立一个中心仓库，先在服务端建立两条分支

master develop

master保存稳定版(production ready)，开发人员平时的代码都提交到develop分支上

开发者的Git分支

张三的Git分支

因为张三和李四同时开发文章系统，所以就有了下面的分支


1 # 张三的Git分支
2 article (local)
3 lisi/article (via git remote add lisi http://lisi-server/lisi.git)
4 origin/master (via git remote add origin http://remote-server/blog.git)
5 origin/develop


李四的Git分支

跟张三类似，不过article server变成了张三的


1 # 李四的Git分支
2 article (local)
3 zhangsan/article (via git remote add zhangsan http://zhangsan-server/zhangsan.git)
4 origin/master (via git remote add origin http://remote-server/blog.git)
5 origin/develop


王五的Git分支

因为只有王五一个人开发评论系统，所以只要一条远程分支就行了


1 # 王五的Git分支
2 comment (local)
3 origin/master (via git remote add origin http://remote-server/blog.git)
4 origin/develop


开发过程

张三和李四

假设张三负责文章的管理，及前端显示。李四负责文章的分类和标签系统。

张三开发完一部分后(n次本地commit)，提交到本地的git server(也就是李四添加的http://zhangsan-server/zhangsan.git)。

李四开发完一部分后，因为要与张三开发的部分合并，所以需要执行一下rebase或merge


1 # 当前在article分支中
2 git rebase zhangsan/article
3 # 提交到本地的git server (也就是张三添加的http://lisi-server/lisi.git)。
4 git push local/article master


这时张三又开发完了一部分，他也会走跟李四一样的流程，rebase & push (如果有冲突，解决之)。

假设由张三统一负责将每日的开发进度提交到remote develop分支上，张三在执行了上面所说的流程后，提交到服务端的develop分支。但这时很可能王五已经把他写的代码提交了，所以要先执行一下rebase


1 git rebase origin/develop git push origin develop


这样张三和李四的代码就都提交到了服务端的develop分支上。王五那边的流程也一样，只是少了本地同步的过程。

开发完毕，进入测试阶段

经过几天的开发，各个功能都已基本完成，下面就要进入测试阶段。具体做法是在服务端新添加一个分支，命名为release，所以这时服务端就有3个分支了


1 master
2 develop
3 release


三位同学都在本地新建一个branch，对应服务端的release分支

git checkout -b release origin/release

分别进行测试，如果发现问题，则执行pull & push


1 # 当前在release分支下
2 git pull origin release
3 git push origin develop


经过几天的测试后，发现没什么问题了，就可以发布稳定版了，假设为0.1


1 git checkout master
2 git rebase origin/release
3 git tag 0.1
4 git push origin master --tags


同时别忘了把release分支上的代码与develop分支合并，保证develop分支上不会有遗留的bug。(没有找到如何直接merge两个服务端branch的方法)


1 git checkout develop
2 git rebase origin/release
3 git push origin develop
4 # 删除服务端的release分支
5 git push origin :release


接下来就可以继续在develop分支上进行开发了。

添加新特性

博客系统需要添加archives功能，这个功能的开发就交给了新来的赵六。此时，在服务端新开一个分支，命名为feature，这时服务端的分支就变成了这样


1 master
2 develop
3 feature/archive


赵六也在本地建立一个archive分支，每天提交到服务端的feature/archive，赵六的Git分支是这样的(不相关的服务端Git已省去)


1 master
2 archive
3 origin/feature/archive


等到这个feature开发完了，与服务端的develop分支执行一下rebase，然后再提交到服务端的develop，这样博客的archive功能就有了


1 git checkout archive
2 git rebase origin/feature/archive
3 git push origin feature/archive
4 # 如果要删除服务端的feature/archive分支的话
5 git push origin :feature/archive


接下来的流程跟上面的一样：开发完毕后，新建release分支，在那上面进行bug修复，修复完毕后，merge到服务端的master和develop

紧急修复漏洞

博客出现了一个xss漏洞，需要尽快修复，这时新建一个hotfix分支(现在本地建，然后提交到服务端)，然后对漏洞进行修复，修复完后，提交到服务端的master和develop


01 git checkout master
02 git pull origin master
03 git checkout -b hotfix
04 # 漏洞修复...
05 # 修复完后
06 git push origin hotfix
07 git checkout master
08 git rebase hotfix
09 git push origin master
10 git push origin develop
